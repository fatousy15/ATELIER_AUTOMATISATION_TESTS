Atelier — Automatisation des tests d’API publiques

Vous allez choisir une API dans la liste public-apis et construire une mini-solution de tests automatisés déployée sur PythonAnywhere : exécution planifiée, métriques QoS et tableau de résultats.

1) Objectifs & livrable final

Objectif Tester automatiquement une API (contrat + robustesse + QoS) et publier un dashboard.
Livrable Un dépôt GitHub + une app Flask sur PythonAnywhere + un historique de runs en SQLite.
Contraintes Tests non destructifs, charge limitée, clés API (si besoin) via Secret Github.

2) Scénario pédagogique (ce que vous allez faire)

  1. Choisir une API (idéalement sans clé) et documenter le contrat (endpoints, paramètres, codes, schéma).
  2. Écrire des tests “as code” : assertions sur le statut HTTP, le JSON, les champs et types attendus.
  3. Ajouter de la robustesse : timeout, retry simple, gestion 429/5xx, validation d’erreurs attendues.
  4. Mesurer la QoS : latence (avg/p95), taux d’erreur, disponibilité du dernier run.
  5. Déployer sur PythonAnywhere et exposer un dashboard lisible.
  6. Planifier l’exécution (tâche planifiée) et conserver l’historique (SQLite).
Important : les tests doivent être non destructifs (pas de suppression/création de données côté fournisseur), et respecter un rythme raisonnable (ex. 1 run / 5 minutes, 20 requêtes max/run).

3) Architecture cible

Repo GitHub
  └── Code Python + tests (Testing as Code)
           ↓ (via Action Github)
PythonAnywhere (Flask)
  ├── /run       (déclenche un run de tests)
  ├── /dashboard (résultats + historique)
  └── SQLite     (stockage des runs)

Structure de projet recommandée

project/
├─ flask_app.py
├─ tester/
│  ├─ client.py        # wrapper HTTP (timeout, mesure latence)
│  ├─ tests.py         # tests (fonctions)
│  └─ runner.py        # exécute + calcule métriques
├─ storage.py          # SQLite: save_run(), list_runs()
├─ templates/
│  └─ dashboard.html
└─ requirements.txt

4) Plan de tests attendu

A. Tests “Contrat” (fonctionnels)

  • HTTP 200/204 attendu
  • Content-Type JSON
  • Champs obligatoires présents
  • Types (string/int/float/bool/list/object)
  • Cas d’entrée invalide → code 400/404/… attendu

B. Robustesse & QoS (non-fonctionnels simples)

  • Timeout (ex. 3s) + 1 retry max
  • Gestion 429 (rate limit) : backoff / attente
  • Latence : avg + p95 sur N appels
  • Taux d’erreur (4xx/5xx) + disponibilité

5) Barème (sur 20 points)

Critère Attendu Points
Choix API + contrat API documentée, endpoints définis, hypothèses claires 2
Qualité des tests Assertions pertinentes, lisibles, couverture minimale (≥ 6 tests) 6
Robustesse Timeout/retry, gestion erreurs, 429/5xx 4
QoS Latence + taux d’erreur + interprétation 4
Restitution Dashboard clair + historique + “dernier run” 4
Bonus (+2) : planification stable (cron/scheduled task), export JSON téléchargeable, endpoint /health, ou visualisation simple (trend latence / erreurs).

Checklist opérationnelle

  • ✅ Repo GitHub créé
  • ✅ Tests implémentés (≥ 6)
  • ✅ Timeout + 1 retry max
  • ✅ Enregistrement des runs (SQLite)
  • ✅ Dashboard accessible (/dashboard)
  • ✅ Exécution planifiée (PythonAnywhere Scheduled Task)
  • ✅ [Bonus] /health pour visualiser l'état de santé de votre solution

Rappels “bonnes pratiques”

  • Ne pas hardcoder une clé API (utiliser les Secrets Github).
  • Limiter la charge (ne pas spammer l’API).
  • Gérer proprement les codes 429, 5xx, timeouts.
  • Tracer les résultats (timestamp, latence, statut, message) dans une base de données.

Rappel : préférez les API “no auth” pour simplifier votre solution. Dans le cas contraire pensez à mettre vos clés dans des variables d’environnement. Prévoyez un mécanisme anti-spam (limite de runs) et un timeout strict (3–5s).
